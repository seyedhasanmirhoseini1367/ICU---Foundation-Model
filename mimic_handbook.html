Introduction

What is MIMIC?

Medical Information Mart for Intensive Care — Beth Israel Deaconess Medical Center, Boston · Managed by MIT Lab for Computational Physiology · Hosted on PhysioNet

MIMIC is one of the largest freely available clinical datasets in the world. It contains de-identified health data from patients admitted to ICUs and inpatient units at a large US academic medical center between 2001 and 2019. It includes demographic data, hourly vital signs, laboratory results, medications, nursing notes, procedures, billing codes, and death records — both in-hospital and post-discharge.

Requires credentialed PhysioNet account and signing a data use agreement. The dataset is free but not public — you cannot share the raw files. De-identified per HIPAA Safe Harbor.

MIMIC-III · 2001–2012 · Flat Schema · UPPERCASE filenames

MIMIC-III Files (26 total)

Single flat folder — no hosp/icu split · 46,520 patients · 61,532 ICU stays · ICD-9 only · Files use UPPERCASE names

⚠ MIMIC-III vs MIMIC-IV key differences
  • All 26 files live in one flat folder — no hosp/ or icu/ split
  • Filenames are UPPERCASE (CHARTEVENTS.csv, PATIENTS.csv…)
  • Only 46,520 patients vs 364,627 in MIMIC-IV
  • Covers 2001–2012 only (MIMIC-IV: 2008–2019)
  • ICD codes are ICD-9 only — MIMIC-IV added ICD-10
  • NOTEEVENTS.csv — ~2M free-text clinical notes. This file does NOT exist in MIMIC-IV and is one of MIMIC-III's most unique features
  • Two separate input tables: INPUTEVENTS_CV (CareVue system) and INPUTEVENTS_MV (MetaVision system) — merged in MIMIC-IV
  • CALLOUT.csv — ICU discharge decisions and delays — no MIMIC-IV equivalent
  • DATETIMEEVENTS.csv — date/time-valued chart events (separate from CHARTEVENTS) — merged in MIMIC-IV
  • Uses CPT billing codes (CPTEVENTS + D_CPT) — MIMIC-IV uses HCPCS instead

Step-by-step plan

Foundation Model Build Roadmap

From raw CSV files to a trained model — ordered by what to do first. Each phase builds on the previous.

Your questions answered directly:
"Should I separate data by subject_id?" — Yes, but not first. You first decide what you want to predict, then filter which patients to include, then build one timeline per patient (by subject_id). The data is spread across files on purpose — the files are the source; your pipeline stitches them together.

"Because they are in a file and some files?" — Correct. A single patient's data lives in chartevents, labevents, inputevents, outputevents — all linked by subject_id (and stay_id for ICU). Your pipeline queries each file for one patient's rows and merges them into one sorted event list.

"What is the first step?" — Phase 1: define your model's unit of input. Decide what one training example looks like before writing any data code. Everything else follows from that decision.
0
Define your goal & input representation
Do this before touching data. These decisions shape everything.
  • What is one training example? — Typically: one ICU stay. All events from admission to discharge (or first 48h) for one patient.
  • What is one token? — One medical event: (Δt from ICU admit, source_table, itemid, value_bin). Not a word — a measurement or drug administration.
  • What is your pre-training objective? Choose one:
    → Next-event prediction (GPT-style): given events 1…N, predict event N+1. More natural for sequences.
    → Masked-event modeling (BERT-style): mask 15% of events, predict what they were. Better at representation learning.
  • What downstream tasks? — In-hospital mortality, 30-day readmission, ICU length-of-stay, sepsis onset. These define what labels you need and how you split data.
  • What window length? — First 24h or 48h is the standard benchmark (most mortality papers use 24h). Full-stay is richer but harder to batch.
1
Cohort selection — who goes in
Filter BEFORE building the dataset. Saves enormous time later.
  • Start from icustays.csv — this is your master list. Each row = one ICU stay = one training example.
  • Filter: ICU length of stay ≥ 24 hours (very short stays have almost no events).
  • Filter: age at ICU admission ≥ 18 years — join with patients.csv to compute age from anchor_age (MIMIC-IV) or dob (MIMIC-III). Exclude paediatric patients unless that is your goal.
  • Filter: must have at least 50–100 chart events — join with chartevents and count per stay_id. Stays with almost no data are not useful.
  • Result: a list of stay_id (MIMIC-IV) or icustay_id (MIMIC-III) values — your cohort. Save this as cohort.csv. Everything downstream is filtered to this list.
  • Typical result from MIMIC-IV: ~50,000–60,000 stays after these filters.
2
Train / Validation / Test split — by patient, not by row
This step must happen before you extract any features, to prevent data leakage.
  • Split by subject_id, not by stay_id or by row. The same patient can have multiple ICU stays — all stays for one patient must go into the same split.
  • Typical split: 70% train / 10% validation / 20% test.
  • Stratify by outcome: ensure similar mortality rates in each split. Patients who died are a minority — random split can leave too few deaths in test.
  • Save three files: split_train.txt, split_val.txt, split_test.txt — each containing a list of subject_ids. Lock these and never change them.
  • All preprocessing and normalisation parameters (means, standard deviations, bin edges) are computed only on the training split, then applied to val/test.
3
Patient-centric extraction — YES, separate by subject_id
The core data engineering step. Query every event table filtered by your cohort.
  • For each stay in your cohort, query four tables and extract their rows:
    • chartevents — filter by stay_id IN (cohort) → vitals, ventilator settings, nurse assessments
    • labevents — filter by hadm_id IN (cohort_admissions) → lab results
    • inputevents — filter by stay_id IN (cohort) → drugs and fluids
    • outputevents — filter by stay_id IN (cohort) → urine output, drain volumes
  • Use DuckDB for this — it can filter 433M chartevents rows in seconds without loading the file into RAM. Query once per table, not per patient.
  • Add a source column to each extracted table: 'CHART', 'LAB', 'INPUT', 'OUTPUT'.
  • Standardise the time column to one name: charttime.
  • Merge all four into one big DataFrame per patient, sorted by charttime.
  • Save as Parquet files partitioned by subject_id: patients/subject_12345.parquet. Or batch into a single Parquet with subject_id as a partition key. Parquet compresses 5–10× better than CSV and loads instantly.
Why Parquet per patient? During training, the DataLoader reads one patient at a time. If all patients are in one giant file, every read scans the whole file. Per-patient Parquet files let you read exactly one patient's 2,000 events in milliseconds.
4
Data cleaning — per item, on the training split only
Remove noise before building the vocabulary.
  • Drop rare itemids: remove any itemid that appears in fewer than 50 patients. These have too little data to learn a reliable representation.
  • Remove physiological outliers: heart rate > 300, SpO2 > 100, negative values where impossible. Use per-itemid percentile bounds (e.g. keep 0.5th–99.5th percentile).
  • Remove duplicate events: same stay_id + itemid + charttime (within 1 minute) → keep the latest storetime.
  • chartevents quality flag: drop rows where warning = 1 (manually flagged as suspicious by nurses).
  • inputevents: drop rows where statusdescription = 'Rewritten' — these are amended orders, not actual administrations.
  • Clip event sequence to your chosen window: if using 48h, drop all events after 48h from intime in icustays. Compute Δt = charttime - intime in hours.
5
Build the vocabulary & tokenise values
Turn continuous numbers into discrete tokens the model can learn from.
  • Vocabulary = set of (source, itemid) pairs. Every distinct item type in every table gets a unique integer ID. Example: (CHART, 220045) → token 42, (LAB, 50912) → token 387.
  • Value discretisation: for each itemid, compute quantile bins on the training split (e.g. 10 bins → 10 possible value tokens). A heart rate of 72 bpm → bin 4 of 10 for itemid 220045. This converts continuous values to categorical and handles outliers automatically.
  • Time encoding: Δt in hours from ICU admission. Do not discretise — feed as a continuous float or use a learned time embedding.
  • One token = one tuple: (item_token_id, value_bin, Δt). This is what goes into the model's embedding layer.
  • Special tokens: add [PAD] (padding), [MASK] (for masked modeling), [SEP] (between days), [ICU_ADMIT], [ICU_DISCHARGE].
  • Save vocab as vocab.json — you need it at inference time to decode predictions back to clinical labels.
6
Model architecture
Start small. Prove the pipeline works, then scale.
  • Input embedding: each event = itemid_embedding + value_embedding + time_embedding. Sum or concatenate these three.
  • Time embedding: use sinusoidal encoding of Δt, or learn a separate embedding. This is critical — ICU data is irregularly sampled (vitals every hour, labs every 6h).
  • Transformer backbone: 4–12 layers, 128–512 hidden dim, 4–8 attention heads. For a first prototype, 4 layers / 128 dim is enough to validate the pipeline.
  • Causal mask (GPT-style): each event attends only to earlier events. Enforces temporal direction.
  • Output head: for next-event prediction → linear layer projecting to vocab size → softmax. For downstream tasks → linear layer projecting to 1 → sigmoid (binary outcome).
  • Sequence length: 512 events covers roughly 6–12h of dense ICU data. Pad shorter sequences, truncate longer ones.
7
Pre-training
Train on the event stream. No labels needed here — the data IS the supervision.
  • Batch construction: one ICU stay = one sequence. Batch 16–64 sequences together. Pad to same length within each batch.
  • Objective: predict the next event's item_token_id and value_bin. Loss = cross-entropy on both. This forces the model to learn: "after a heart rate of 90 and a norepinephrine dose, what comes next?"
  • Optimizer: AdamW, learning rate ~1e-4 with warmup (first 1,000 steps), cosine decay.
  • Monitor: validation perplexity (lower = better). Should drop quickly in the first few epochs.
  • Compute: on MIMIC-IV (~50k stays × 512 events), a 4-layer model trains in a few hours on a single GPU. A 12-layer model takes 1–2 days.
  • Checkpoint: save the best model by validation loss. This is your pre-trained foundation model.
8
Fine-tuning & evaluation
Apply the pre-trained model to specific clinical prediction tasks.
  • Freeze the backbone, add a task-specific head (linear layer), train for 5–20 epochs on each task. Or fine-tune the whole model end-to-end if you have enough labelled data.
  • Task 1 — In-hospital mortality: label = hospital_expire_flag from admissions.csv. Standard benchmark: AUROC. Target: ≥0.85 is competitive.
  • Task 2 — ICU length of stay: label = los from icustays.csv. Treated as regression (MAE) or binary (>3 days).
  • Task 3 — 30-day readmission: label derived from admissions.csv (is there another admission within 30 days of discharge?).
  • Baselines to beat: LightGBM on hand-crafted features, LSTM, RETAIN. These are the standard comparisons in MIMIC papers.
  • Report on the test split only — the one that was locked in Phase 2 and never touched during training.
9
Scale up & federate (PhD scope)
Once the pipeline works on MIMIC, extend to multi-site data.
  • Problem: hospitals use different itemids for the same measurement. Amsterdam UMC's "heart rate" has a different code than BIDMC's. You need a concept mapping layer: map site-specific itemids to a shared concept vocabulary.
  • Federated learning: each hospital trains a local copy of the model on their data, only model weights (not patient data) are shared. A central server aggregates weights (FedAvg). No patient leaves their hospital.
  • Site-specific tokens: add a [SITE_X] token at the start of each sequence so the model learns site-level differences without confounding the clinical signal.
  • MIMIC-III + MIMIC-IV together: MIMIC-III covers 2001–2012, MIMIC-IV covers 2008–2019 (overlapping). If combining, deduplicate patients (same subject_id appears in both) and align itemids using D_ITEMS.
  • This is exactly the GPT-MEDIC PhD problem: generalise across Amsterdam, UCL, Cambridge — each with ~300k patients — while respecting data-sharing restrictions.

What this file teaches you about data organisation

Why delta_hours and not raw timestamps

The model needs to learn relative timing, not absolute dates. A blood pressure drop 2 hours after a drug is the same pattern whether it happened in 2178 or 2008. delta_hours from ICU admission makes patterns comparable across patients.

Negative delta_hours — pre-ICU events

Labs drawn in the ED before ICU transfer have negative delta. These are valuable — they show the patient's baseline before critical illness. Keep them. The model can learn: "WBC=17 before admission → bad outcome".

One row = one token

Each row in this merged file becomes one token in the model's input sequence: (Δt, source, itemid, value). 6,923 rows = 6,923 tokens for this one patient. The model reads them left to right, in time order.

Multiple ICU stays = multiple sequences or one

This patient has 3 stays. You can treat each stay as an independent training example, or concatenate them with a [STAY_SEP] special token between them. Most papers treat each stay independently.
